# AGENT.md — правила проекта «Учёт занятий» (LessonTracker)

> Этот файл — «конституция» проекта для AI-агента (вайбкодинг) и для людей.
> **Читать в начале КАЖДОЙ сессии.** Не переизобретать архитектуру и стек — они зафиксированы ниже.

## 1. Цель проекта
Мобильное приложение (Android/APK) для преподавателя: учёт проведённых занятий,
посещаемости и успеваемости по предметам и группам.

**Единица учёта — ЗАНЯТИЕ**, не академический и не астрономический час.
У предмета есть план в занятиях; контролируем проведённые / оставшиеся занятия.

## 2. Стек (НЕ менять без явного согласования с владельцем)
- Язык: **Python 3**
- UI: **Kivy** (чистый Kivy; диаграммы рисуем средствами `canvas`, БЕЗ matplotlib)
- БД: **SQLite** (модуль `sqlite3` из стандартной библиотеки)
- Сборка APK: **только GitHub Actions** (см. §6). Локально APK НЕ собираем.

### Запрещённые зависимости
- ❌ **matplotlib, numpy, pillow** — тяжёлые нативные колёса, основная причина
  вылета APK «не запустившись». Диаграммы — только средствами Kivy canvas.
- Любая новая зависимость добавляется только после согласования и с фиксацией версии.

## 3. Архитектура (модульная, весь SQL изолирован)
- `database.py` — ВЕСЬ доступ к SQLite (единственное место с SQL). Миграция на другую
  СУБД = переписать только этот файл, сохранив сигнатуры функций.
- `models.py` — датаклассы/структуры данных, без SQL и без UI.
- `main.py` — точка входа Kivy-приложения и экраны.
- `charts.py` — отрисовка диаграмм на Kivy canvas (без сторонних либ).
- `calendar_export.py` — выгрузка расписания в `.ics` (Google/Яндекс), без UI.
- `tests/` — автотесты «чистой» логики (БД, .ics). UI автотестами НЕ покрываем.
- `docs/` — документация. `docs/ROADMAP.md` — план работ.
- `seed_db.py` — создание и наполнение тестовой БД `lessons.db`.

## 4. Definition of Done (для любого среза)
1. Код запускается на ПК без ошибок.
2. Автотесты «чистой» логики зелёные (`pytest`).
3. Изменение закоммичено; рабочая веха помечена git-тегом.
4. Документация обновлена, если менялось поведение.

## 5. Правила процесса (важнее кода)
- **Заморозка:** работающий модуль НЕ рефакторим без падающего теста.
  Принцип «работает → коммит → не трогаем».
- **Вертикальные срезы:** доводим один пункт ROADMAP до рабочего состояния и коммитим,
  прежде чем начинать следующий. Не «всё сразу».
- **Ручная проверка на каждом этапе:** каждый срез перед коммитом запускается ЛОКАЛЬНО
  на ПК и проверяется человеком. Только после подтверждения — commit/push. Агент не
  «идёт дальше» сам, пока человек не подтвердил, что этап работает.
- **Фиксируем маленькие победы:** каждый рабочий срез — отдельный коммит; вехи — теги
  (`v0.1-db-ok`, `v0.2-ui-ok`, ...). Всегда есть куда откатиться.
- **Backlog, а не реактивность:** работаем по `docs/ROADMAP.md` сверху вниз.
- При первом краше на устройстве — обязательно снять `adb logcat`, а не гадать.

### Работа с репозиторием
- Папка уже связана с GitHub. Обмен — только через **commit/push** (ZIP-архивы не делаем).
- Тестовая БД `lessons.db` создаётся скриптом `seed_db.py` и **коммитится в репозиторий**
  (по требованию владельца — база с тестовыми данными хранится в git).

## 6. Сборка APK (CI)
- Триггер: push / commit в репозиторий.
- Тип: **debug-APK** (без подписи, ставится вручную).
- Результат: `.apk` публикуется как artifact сборки GitHub Actions.
- Workflow: `.github/workflows/build-apk.yml`. Версии инструментов пинуются.

## 7. Что НЕ трогать
- Публичные сигнатуры функций в `database.py` (на них завязан UI и тесты).
- Стек и запрещённые зависимости из §2.
- Принцип «единица учёта — занятие».
